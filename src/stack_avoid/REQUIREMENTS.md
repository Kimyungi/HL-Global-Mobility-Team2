# stack_avoid — 요구사항

**담당: 이기돈** · 산출물: 회피 단독 주행 (8/2) — 하위 제어 보수 겸임(김재민에게서 인수)

## 역할

장애물 인지, 회피 가능 판정 재료(TTC·측방), 회피 경로

**시나리오:** 주행 중 전방에 장애물 → 회피 경로로 피해서 지나간 뒤 원래 경로로 복귀.

**구현 방식: 실시간 스캔 기반 반응형으로 구현한다 (맵 생성 불필요).**
스캔은 본질적으로 차량 중심 좌표라 vehicle frame 변환이 거의 공짜고, ref points가
매 주기 재생성되는 아키텍처(CLAUDE.md §7)와도 정확히 맞는다. 주차용 로컬맵은 stack_parking 소관.

## 계약 (이것만 지키면 나머지는 자유)

- 입력: 2D LiDAR ×2 (100ms).
- 출력: `/perception/avoid` (`fma_interfaces/AvoidStatus`).
  - `avoidable`(TTC·측방 여유)은 판정 **재료** — lane/waypoint→avoid 전이 결정은 MGM.
  - `ttc`: 장애물이 없으면 반드시 큰 값(1e9). 0으로 두면 MGM이 즉시 정지 바닥을 밟는다.
    계산은 정적 장애물 가정(거리 ÷ 자차 속도)으로 충분.
  - `points[]`: vehicle frame 회피 경로, `maneuver_done`: 복귀 트리거.
  - `v_suggest`: 회피 기하로 결정한 속도 — avoid 스테이트의 종방향은 이 값이 지배.
- **`maneuver_done` 주의 — 스캔 기반의 유일한 함정.** 장애물을 지나치는 중에는 장애물이
  전방 스캔 시야에서 사라지지만 아직 차 **옆에** 있다. "스캔에 안 보임 = 완료"로 구현 금지.
  측방 클리어런스 확인이나 최근 장애물 위치의 짧은 유지·감쇠 같은 내부 상태는 허용
  (이건 맵 생성이 아니라 정상적인 내부 구현이다).
- **복귀 경로를 정밀하게 계획할 필요 없음** — 회피 경로는 "비켜 가는 것"까지만 책임.
  `maneuver_done`이 서면 MGM이 스테이트를 복귀시키고 차선/GPS 경로가 다시 소스가 되며,
  조립 블록의 블렌드가 전환 불연속을 흡수한다.
- **회피 불가 케이스는 이 스택 책임이 아님.** `avoidable=false`로만 두면 된다 — 그 경우의
  정지는 stack_estop(박찬미)의 TTC 기준이 커버한다. 두 스택이 동시에 반응해도 안전
  (우선권 표에서 estop이 최상위). 역할 경계: **avoid는 회피 성립만, estop은 위험 임박만 본다.**
- 금지: 이 스택에서 정지/모드 결정 금지. "회피 경로 요구" 같은 별도 요구 개념도 없음 — 회피는 스테이트다 (CLAUDE.md §4).

## 공통 규칙 (CLAUDE.md)

- 출력은 `fma_interfaces` 메시지로만. MGM은 이 토픽만 구독한다.
- 경로를 내는 스택은 전부 동일 ref points 포맷 — {x, y, yaw, curvature}, vehicle frame (§5.4).
- 판단 로직(모드 전환·정지 결정·우선권)은 MGM 스테이트 머신에만 존재한다 (§4, §5.1).
- 실행: `ros2 run stack_avoid stack_avoid_node`
